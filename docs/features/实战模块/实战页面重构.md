## 1. 背景与价值

**为什么做**：`battle_screen.dart` 文件超过 4500 行代码，承担状态管理、业务逻辑、UI 渲染等多重职责，严重违反单一职责原则（SRP）。代码难以维护、难以测试、功能扩展困难。

**现状问题**：
- 40+ 个 State 变量分散管理
- 87 个方法职责混杂
- K线处理、交易计算、选股逻辑混合
- 11 个指标图表独立实现但代码重复
- 数据库操作内嵌违反分层架构

**做成什么样**：
- 文件拆分：将 4500+ 行文件拆分为 10-15 个职责单一的文件
- 状态集中：使用 Riverpod 统一管理训练状态
- 逻辑解耦：业务逻辑与 UI 分离，提升可测试性
- 代码复用：抽取公共组件和逻辑
- 保证原有页面格局和功能不变

**关联**：
- K线训练营 App 核心功能优化
- 涉及首页、实战、记录、我的四个 Tab 页

---

## 2. 范围

### 本次做

- **架构重构**：将 BattleScreen 拆分为 Provider/Service/Model/Widget 多层架构
- **状态管理**：使用 Riverpod 统一管理训练状态
- **UI组件拆分**：将 50+ 个 Widget 构建方法抽取为独立组件
- **缓存机制**：添加指标计算缓存，避免重复计算
- **状态恢复**：实现 Provider 内存缓存 + 数据库持久化的混合方案
- **数据一致性**：统一股票代码、日期时间等字段格式
- **错误处理**：建立统一的异常处理机制
- **日志监控**：添加性能监控和日志记录
- **UI规范**：统一字体大小、溢出处理、颜色规范

### 本次不做

- 不修改现有数据库表结构
- 不添加新的功能模块
- 不修改其他页面（首页、记录、我的等）
- 不修改 K线图表渲染逻辑
- 不修改交易功能核心逻辑
- 不修改 11 个指标图表的计算逻辑

## 4. 需求详述

### 4.1 核心流程

#### 流程A：直接进入实战页面（随机选股训练）

```
1. 用户点击底部导航栏"实战"
       ↓
2. 系统检查是否有路由参数
       ↓
3. 如果无路由参数：
   - 查询数据库获取有至少 210 天日线数据的股票列表
   - 随机选择一只合格股票
   - 加载该股票的 K线数据
       ↓
4. 进入训练界面，显示 K线图表和随机股票信息
       ↓
5. 用户可以进行 K线步进、买入/卖出等操作
```

#### 流程B：从首页选股后跳转实战页面

```
1. 用户在首页选择选股条件和时间范围
       ↓
2. 点击确认按钮，携带参数跳转到实战页面
       ↓
3. 实战页面接收路由参数（symbol、name、market、date 等）
       ↓
4. 加载指定的股票 K线数据
       ↓
5. 进入训练界面，显示指定股票信息
```

#### 流程C：Tab 切换后状态恢复

```
1. 用户在实战页面进行训练
       ↓
2. 用户点击底部导航栏切换到其他 Tab
       ↓
3. IndexedStack 隐藏实战页面（但保留 Provider 状态）
       ↓
4. 用户点击"实战"Tab 返回
       ↓
5. IndexedStack 显示实战页面
       ↓
6. 实战页面从 Provider 恢复所有训练状态
```

#### 流程D：复盘模式

```
1. 用户从记录页面选择历史训练
       ↓
2. 系统通过 sessionId 加载历史训练数据
       ↓
3. 实战页面进入复盘模式
       ↓
4. 用户可以查看历史 K线数据和交易记录
       ↓
5. 交易按钮禁用，仅供查看
```

### 4.2 交互规则

- 点击 Tab 项立即切换页面，无动画过渡
- 当前 Tab 显示高亮状态，非当前 Tab 显示普通状态
- 返回时立即显示恢复后的状态，无加载过程
- 复盘模式下买入/卖出按钮禁用，显示"已结束"
- 训练周期边界时显示训练结束弹窗
- 数据不足时显示友好提示，不使用模拟数据

### 4.3 业务规则

| 编号 | 规则 |
|------|------|
| BR-001 | 随机选股仅从数据库中有至少 210 天日线数据的股票中选择 |
| BR-002 | 训练周期固定为 150 天 |
| BR-003 | 前置 K线显示 60 天用于技术指标预热 |
| BR-004 | 初始资金固定为 100000 元 |
| BR-005 | Tab 切换必须保持所有训练状态不丢失 |
| BR-006 | 所有 K线数据必须从数据库真实获取，禁止使用模拟数据 |
| BR-007 | 股票代码格式统一：前端使用市场前缀（如 SH600000），数据库使用对应格式 |
| BR-008 | 日期时间格式统一：存储使用 ISO8601，显示使用 yyyy-MM-dd |
| BR-009 | 涨跌颜色：红涨绿跌 |
| BR-010 | 指标计算使用缓存机制，避免重复计算 |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-FUNC-001**: 随机选股初始化
  - **Given** 用户直接点击底部导航栏"实战"进入实战页面
  - **When** 页面首次加载且无路由参数
  - **Then** 系统从数据库随机选择一只至少有 210 天日线数据的股票并加载 K线数据

- [ ] **AC-FUNC-002**: K线步进功能
  - **Given** 实战页面加载了 K线数据
  - **When** 用户点击"下一步"
  - **Then** K线推进一天，账户状态根据收盘价更新，显示"收盘"标记

- [ ] **AC-FUNC-003**: 买入/卖出功能
  - **Given** 实战页面有可用资金或持仓
  - **When** 用户点击"买入"或"卖出"并完成操作
  - **Then** 账户资金和持仓更新，显示交易记录点

- [ ] **AC-FUNC-004**: 指标图表显示
  - **Given** 实战页面加载了 K线数据
  - **When** 用户切换指标类型（MACD/KDJ/RSI/BOLL 等）
  - **Then** 对应的指标图表正确显示，数据与 K线对齐

- [ ] **AC-FUNC-005**: 复盘模式
  - **Given** 用户从记录页面跳转进入实战页面并带有 sessionId
  - **When** 页面加载
  - **Then** 系统加载历史训练数据，买入/卖出按钮禁用，可查看历史交易

- [ ] **AC-FUNC-006**: Tab 切换状态恢复
  - **Given** 用户在实战页面进行训练，已进行 10 天 K线步进
  - **When** 用户点击"记录"Tab 切换到记录页面，再点击"实战"返回
  - **Then** 实战页面显示第 10 天的状态，K线位置、资金、持仓、交易记录均保持

- [ ] **AC-ROUTER-001**: 从首页选股跳转
  - **Given** 用户在首页选择股票后点击确认
  - **When** 跳转到实战页面
  - **Then** 实战页面正确加载指定股票的数据

- [ ] **AC-ROUTER-002**: 复盘模式路由
  - **Given** 用户从记录页面选择历史训练
  - **When** 点击复盘按钮
  - **Then** 系统通过 sessionId 正确加载历史数据

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-ERROR-001**: 数据库无合格股票
  - **Given** 数据库中所有股票数据都不足 210 天
  - **When** 执行随机选股
  - **Then** 显示提示"数据库中暂无合格股票，请先同步数据"，不使用模拟数据

- [ ] **AC-ERROR-002**: 数据加载失败
  - **Given** 网络或数据库异常
  - **When** 加载 K线数据
  - **Then** 显示友好提示"数据加载失败，请检查网络后重试"，显示重试按钮

- [ ] **AC-ERROR-003**: 交易失败 - 余额不足
  - **Given** 账户余额不足以买入
  - **When** 用户尝试买入
  - **Then** 显示错误提示"可用余额不足，无法买入"，阻断交易操作

- [ ] **AC-ERROR-004**: 交易失败 - 无持仓
  - **Given** 当前无持仓
  - **When** 用户尝试卖出
  - **Then** 显示错误提示"当前无持仓，无法卖出"，阻断交易操作

- [ ] **AC-DATA-005**: 无合格股票时的处理
  - **Given** 随机选股查询后无符合条件股票
  - **When** 执行初始化
  - **Then** 显示"数据库中暂无合格股票，请先同步数据"，不使用保底股票或模拟数据

- [ ] **AC-UI-005**: 文本溢出处理
  - **Given** 股票名称过长或价格数字过长
  - **When** 页面渲染
  - **Then** 使用溢出处理（ellipsis 或 FittedBox），不出现溢出错误

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-ARCH-001**: 架构 - 文件行数控制
  - **Given** 重构完成后
  - **When** 检查 battle_screen.dart 文件行数
  - **Then** 文件行数少于 600 行

- [ ] **AC-ARCH-002**: 架构 - 状态集中管理
  - **Given** 重构完成后
  - **When** 检查代码结构
  - **Then** 所有训练状态变量迁移到 BattleProvider，State 变量列表为空

- [ ] **AC-ARCH-003**: 架构 - Widget 独立可编译
  - **Given** 重构完成后
  - **When** 分别编译各 Widget 文件
  - **Then** 各 Widget 可独立编译通过

- [ ] **AC-ARCH-004**: 架构 - Service 层可测试
  - **Given** 重构完成后
  - **When** 对 Service 层编写单元测试
  - **Then** StockService、TrainingService 可独立测试

- [ ] **AC-ARCH-005**: 架构 - 指标缓存机制
  - **Given** 指标计算后
  - **When** 再次访问同一指标
  - **Then** 使用缓存数据，避免重复计算

- [ ] **AC-ARCH-006**: 架构 - 状态持久化
  - **Given** 用户进行训练操作
  - **When** 点击下一步或执行交易
  - **Then** 训练状态保存到数据库，支持跨会话恢复

- [ ] **AC-DATA-001**: 数据 - 股票代码格式统一
  - **Given** 股票代码在 UI、Provider、数据库之间传递
  - **When** 保存或加载数据
  - **Then** 使用统一的格式转换（SymbolConverter），以数据库格式为准

- [ ] **AC-DATA-002**: 数据 - 日期时间格式统一
  - **Given** 日期时间在 UI、Provider、数据库之间传递
  - **When** 保存或加载数据
  - **Then** 使用统一的格式转换（DateConverter），存储使用 ISO8601

- [ ] **AC-DATA-003**: 数据 - 以数据库为准
  - **Given** 前后端及数据库可能存在同一字段格式不同
  - **When** 数据在各层之间传递
  - **Then** 以数据库存储格式为准，前端和后端做好格式转换

- [ ] **AC-DATA-004**: 数据 - 路由参数格式
  - **Given** 通过路由传递参数
  - **When** 传递日期时间等参数
  - **Then** 使用存储格式（ISO8601）

- [ ] **AC-DATA-006**: 数据 - 无模拟数据
  - **Given** 数据库无合格股票
  - **When** 系统尝试加载数据
  - **Then** 显示提示"数据不足，无法进行训练"，不存在静默生成的测试数据

- [ ] **AC-UI-006**: UI - 按钮文字缩放
  - **Given** 按钮文字过长（如"已结束"）
  - **When** 页面渲染
  - **Then** 使用 FittedBox 缩放，不出现溢出

- [ ] **AC-UI-007**: UI - 涨跌颜色
  - **Given** K线数据显示涨跌
  - **When** 价格相比昨日收盘价上涨
  - **Then** 显示红色（Colors.red）
  - **When** 价格相比昨日收盘价下跌
  - **Then** 显示绿色（Colors.green）
