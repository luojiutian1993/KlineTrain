# 指标预加载与缓存优化需求

## 1. 背景与价值

### 1.1 为什么做

当前实战页面存在以下问题：
1. **指标显示断档**：MACD、KDJ等技术指标以及5日线、10日线、30日线在图表最左侧显示为横线或空白
2. **数据计算错误**：使用日历天计算，而非工作日（交易日），导致加载的数据量不足
3. **指标数据缺失**：均线和指标的预计算不完整，缺少前置预热数据

**根本原因**：
- 股票交易按工作日（交易日）计算，不是按日历天计算
- 150个工作日 ≈ 210个日历天
- 100个工作日 ≈ 140个日历天
- 33个工作日 ≈ 46个日历天

### 1.2 做成什么样

实现三层数据加载策略，通过以下方式确保用户缩放或左滑到最左侧时，所有技术指标都能无断档地正常显示：
- 使用工作日（交易日）精确计算数据加载范围
- 节假日表精确计算工作日对应的日历天数
- 三层数据加载：训练周期 + 预加载数据 + 指标前置数据
- 数据不充足时自动切换到数据充足的股票
- 前端用户展示的数据始终是足量数据

### 1.3 关联

- 现有指标计算逻辑（`indicator_calculator.dart`）
- 现有预计算逻辑（`battle_provider.dart`）
- K线图表交互优化需求文档

## 3. 三层数据加载策略

### 3.1 数据分层结构

```
┌────────────────────────────────────────────────────────────────────────────┐
│ 第四层：后续数据（额外20天缓冲）                                              │
│ 这部分数据不展示给用户，仅用于确保数据查询结果满足前端需求                         │
│                                                                            │
│ ┌────────────────────────────────────────────────────────────────────┐    │
│ │ 第三层：训练日期范围（150个工作日）                                          │    │
│ │ 训练范围：索引 133 ~ 282（共150天）                                       │    │
│ │ 用户主要训练区域，点击"下一步"推进                                           │    │
│ │ 训练结束后不能继续推进                                                    │    │
│ │                                                                            │    │
│ │ ┌──────────────────────────────────────────────────────────────┐    │    │
│ │ │ 第二层：预加载数据（100个工作日，支持左滑和缩放展示）                      │    │    │
│ │ │ 展示范围：索引 33 ~ 132（133天）                                    │    │    │
│ │ │ 用于支持用户左滑查看历史数据和指标预热                                  │    │    │
│ │ │                                                                            │    │    │
│ │ │ ┌────────────────────────────────────────────────────────┐    │    │    │
│ │ │ │ 第一层：指标计算前置数据（33个工作日）                          │    │    │    │
│ │ │ │ 索引：0 ~ 32（33天）                                      │    │    │    │
│ │ │ │ 用于计算预加载数据前33天的MACD、均线等指标                    │    │    │    │
│ │ │ └────────────────────────────────────────────────────────┘    │    │    │
│ │ └──────────────────────────────────────────────────────────────┘    │    │
│ └────────────────────────────────────────────────────────────────────┘    │
│                                                                            │
│ 最终加载范围：约343个日历天（对应303个工作日 = 283训练需求 + 20缓冲）             │
└────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 索引计算公式

| 变量 | 计算方式 | 说明 |
|------|---------|------|
| **训练起点索引** | `initialStartIndex = 133` | 固定值：100（预加载） + 33（指标前置） |
| **训练终点索引** | `trainingEndIndex = initialStartIndex + trainingDays - 1` | 训练起点 + 训练天数 - 1 = 282 |
| **训练天数** | `trainingDays = 150` | 固定配置值 |
| **指标前置数据** | `indicatorPreloadDays = 33` | MACD指标最大前置需求 |
| **预加载数据** | `preloadDays = 100` | 支持左滑查看历史 |
| **缓冲数据** | `bufferDays = 20` | 额外查询，用于数据充足性保障 |

### 3.3 数据使用规则

| 数据类型 | 索引范围 | 工作日 | 使用场景 | 是否展示 |
|---------|---------|--------|---------|---------|
| **指标前置数据** | 0 ~ 32 | 33天 | 计算MACD等指标的前置数据 | ✅ 展示（作为历史的一部分） |
| **预加载数据** | 33 ~ 132 | 100天 | 左滑查看历史，指标预热 | ✅ 展示 |
| **训练数据** | 133 ~ 282 | 150天 | 用户点击下一步推进 | ✅ 展示 |
| **后续数据** | 283 ~ (length-1) | ~20天 | 数据充足性保障 | ❌ 不展示 |

**重要说明**：
- **额外查询的20天缓冲数据**不用于训练，也不用于历史展示，仅用于确保数据查询结果充足
- 后续数据（训练终点之后的数据）同样不展示给用户，仅作为数据充足性保障
- 训练终点索引 `trainingEndIndex = initialStartIndex + trainingDays - 1 = 282`
- 到达训练终点后，用户不能继续点击"下一步"，应提示"训练结束"

### 3.3 指标前置数据需求

| 指标 | 公式 | 最小前置天数 | 说明 |
|------|------|-------------|------|
| **MA5** | 简单移动平均 | 5天 | 最近5天收盘价均值 |
| **MA10** | 简单移动平均 | 10天 | 最近10天收盘价均值 |
| **MA30** | 简单移动平均 | 30天 | 最近30天收盘价均值 |
| **MACD** | EMA(12,26,9) | 33天 | EMA(12)需要12天，EMA(26)需要26天，DEA需要9天，取最大值 |
| **KDJ** | RSV(9,3,3) | 9天 | RSV需要9天，K/D/J各需3天平滑 |
| **RSI** | RSI(14) | 14天 | 14天周期 |
| **BOLL** | MA(20)+2*SD(20) | 20天 | 布林带中轨是20日均线 |
| **DMI** | ADX(14) | 14天 | 需要14天计算基础DM值 |

**最大前置需求**：**33天**（MACD指标）

### 3.4 数据加载策略

**核心原则**：
1. **精确计算**：使用工作日（交易日）计算，而非日历天
2. **真实数据**：前置的K线数据必须从数据库真实获取，指标在完整数据上真实计算
3. **自动切换**：数据不充足时自动切换到数据充足的股票

**数据加载流程**：
```
1. 确定训练结束日期（当前日期）
2. 计算需要的工作日总数：150 + 100 + 33 = 283个工作日
3. 查询节假日表，精确计算283个工作日对应的日历天数（约320天）
4. 计算数据加载起始日期：当前日期 - 320天
5. 从数据库查询数据
6. 检查数据是否充足（≥ 283个工作日）
7. 数据不充足？→ 自动切换到其他股票
8. 数据充足？→ 在完整数据上预计算指标
9. 更新状态并显示K线图和指标
```

**前置填充策略**（仅在数据不足时使用）：
- 如果加载的K线数据不足指标计算所需的前置天数（如用户左滑到数据库最早日期）
- 系统才会使用0填充不足的部分
- 这是边界情况，正常情况下所有指标都是真实计算的

## 5. 精确计算工作日

### 5.1 工作日计算逻辑

```dart
class TradingDayCalculator {
  final HolidayDao _holidayDao;

  /// 计算从结束日期往回N个交易日对应的日历天数
  Future<int> tradingDaysToCalendarDays(int tradingDays, DateTime endDate) async {
    int calendarDays = 0;
    int tradingDayCount = 0;
    DateTime currentDate = endDate;

    while (tradingDayCount < tradingDays) {
      final isHoliday = await _holidayDao.isHoliday(currentDate);

      if (!isHoliday) {
        tradingDayCount++;
      }

      calendarDays++;
      currentDate = currentDate.subtract(const Duration(days: 1));
    }

    return calendarDays;
  }
}
```

### 5.2 数据充足性检查

```dart
class DataSufficiencyChecker {
  final HolidayDao _holidayDao;

  Future<DataSufficiencyResult> checkSufficiency({
    required List<KlineModel> data,
    required int requiredTradingDays,
  }) async {
    if (data.isEmpty) {
      return DataSufficiencyResult(
        isSufficient: false,
        reason: '无数据',
        availableDays: 0,
        requiredDays: requiredTradingDays,
      );
    }

    final firstDate = DateTime.fromMillisecondsSinceEpoch(data.first.timestamp);
    final lastDate = DateTime.fromMillisecondsSinceEpoch(data.last.timestamp);

    // 计算数据覆盖的实际交易日数
    final actualTradingDays = await _holidayDao.countTradingDays(firstDate, lastDate);

    return DataSufficiencyResult(
      isSufficient: actualTradingDays >= requiredTradingDays,
      reason: actualTradingDays >= requiredTradingDays ? '数据充足' : '数据不足',
      availableDays: actualTradingDays,
      requiredDays: requiredTradingDays,
    );
  }
}
```

## 6. 自动切换股票逻辑

### 6.1 股票选择策略

```dart
class StockSelector {
  Future<StockSelectionResult> selectSufficientStock({
    DateTime? preferredStartDate,
    required int totalRequiredDays,
  }) async {
    // 获取所有股票
    final symbols = await _repository.getAllSymbols();

    for (final symbol in symbols) {
      // 加载股票数据
      final data = await _loadKlineDataForSymbol(symbol, preferredStartDate);

      // 检查数据充足性
      final sufficiencyCheck = await _sufficiencyChecker.checkSufficiency(
        data: data,
        requiredTradingDays: totalRequiredDays,
      );

      if (sufficiencyCheck.isSufficient) {
        return StockSelectionResult(
          symbol: symbol,
          data: data,
          sufficiencyCheck: sufficiencyCheck,
          isAutoSelected: preferredStartDate == null,
        );
      }
    }

    // 没有找到数据充足的股票
    return StockSelectionResult(
      symbol: null,
      data: [],
      sufficiencyCheck: null,
      isAutoSelected: false,
      error: '没有找到数据充足的股票',
    );
  }
}
```

### 6.2 切换逻辑集成

在 `initializeWithSymbol` 方法中集成自动切换逻辑：

```dart
Future<void> initializeWithSymbol({
  required String symbol,
  String? name,
  String? marketCode,
  DateTime? startDate,
}) async {
  // 1. 加载数据并检查充足性
  var klineData = await _loadKlineData(symbol, startDate);
  var currentSymbol = symbol;

  // 2. 检查数据充足性
  final sufficiencyCheck = await _sufficiencyChecker.checkSufficiency(
    data: klineData,
    requiredTradingDays: totalRequiredDays,
  );

  // 3. 数据不充足？自动切换
  if (!sufficiencyCheck.isSufficient) {
    final stockSelection = await _stockSelector.selectSufficientStock(
      preferredStartDate: startDate,
      totalRequiredDays: totalRequiredDays,
    );

    if (stockSelection.symbol != null) {
      // 仅在日志中输出，不显示给用户
      if (stockSelection.isAutoSelected) {
        print('ℹ️ [BattleProvider] 随机选择了数据充足的股票');
      } else {
        print('ℹ️ [BattleProvider] 原股票数据不足，已自动切换到 ${stockSelection.symbol}');
      }

      currentSymbol = stockSelection.symbol!;
      klineData = stockSelection.data;
    } else {
      state = state.copyWith(
        isLoading: false,
        hasAvailableData: false,
        errorMessage: '没有找到数据充足的股票，请同步更多数据',
      );
      return;
    }
  }

  // 4. 继续初始化流程...
}
```

## 7. 按需加载策略

### 7.1 动态加载逻辑

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 用户进入实战页面                                          │
│    ↓                                                        │
│ 2. 精确计算需要的工作日总数                                   │
│    训练周期（150天）+ 预加载（100天）+ 指标前置（33天）        │
│    总计：283个工作日                                         │
│    ↓                                                        │
│ 3. 查询节假日表，计算需要的日历天数                            │
│    283个工作日 ≈ 320个日历天                                 │
│    ↓                                                        │
│ 4. 加载数据并检查充足性                                       │
│    数据充足？→ 预计算指标并显示                                │
│    数据不足？→ 自动切换股票                                    │
│    ↓                                                        │
│ 5. 用户缩放或左滑                                           │
│    ↓                                                        │
│ 6. 计算新数据范围                                           │
│    newStart = min(当前可见起始日期 - 33天指标前置, 数据库最早)  │
│    newEnd = 训练结束日期                                      │
│    ↓                                                        │
│ 7. 检查缓存                                                 │
│    缓存命中？→ 使用缓存                                       │
│    缓存未命中？→ 查询数据库 → 预计算指标 → 更新缓存           │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 前端缓存机制

```
缓存键策略：
- 缓存key = "{symbol}_{dataLength}"
- 例如：600519.XSHG_783

缓存淘汰策略：
- LRU缓存，容量50个
- 相同数据长度使用相同缓存
- 缓存有效期：当前会话内有效

缓存更新策略：
- 用户缩放导致数据长度变化时，加载新数据并更新缓存
- 不持久化到本地存储
```

## 8. 验收标准

### 8.1 数据加载

- [ ] **AC-001**: Given 用户进入实战页面，When 系统加载K线数据，Then 精确计算需要的日历天数（查询节假日表）
- [ ] **AC-002**: Given 用户缩放到700根K线，When 系统加载数据，Then 自动加载700根K线 + 33天指标前置数据
- [ ] **AC-003**: Given 用户缩放到100根K线，When 系统加载数据，Then 自动加载100根K线 + 33天指标前置数据

### 8.2 数据充足性

- [ ] **AC-004**: Given 数据充足，When 系统计算训练起始索引，Then 训练日索引正确计算
- [ ] **AC-005**: Given 数据不足，When 系统加载数据，Then 自动切换到数据充足的股票
- [ ] **AC-006**: Given 所有股票数据都不足，When 用户选择股票，Then 显示错误提示

### 8.3 指标显示

- [ ] **AC-007**: Given K线图缩小或左滑到最左侧，When 用户查看指标，Then MA5、MA10、MA30从最早可见K线开始正常显示曲线（前33天可能显示为0或折线）
- [ ] **AC-008**: Given K线图缩小或左滑到最左侧，When 用户查看指标，Then MACD指标从最早可见K线开始正常显示（不是横线或空白）
- [ ] **AC-009**: Given K线图缩小或左滑到最左侧，When 用户查看指标，Then KDJ、RSI、布林带等其他指标也从最早可见K线开始正常显示
- [ ] **AC-010**: Given 用户在任意缩放级别左滑到最左侧，When 查看指标，Then 所有指标都无断档地正常显示

### 8.4 缓存机制

- [ ] **AC-011**: Given 相同股票第二次缩放到相同级别，When 系统加载数据，Then 使用缓存数据，不重复查询数据库
- [ ] **AC-012**: Given 不同股票进入训练，When 系统加载数据，Then 清除旧缓存，加载新数据
- [ ] **AC-013**: Given 缓存数据达到50个，When 新增缓存，Then 自动清除最旧的缓存项

### 8.5 性能要求

- [ ] **AC-014**: Given 首次加载数据，When 系统查询数据库，Then 响应时间 < 300ms（283天数据）
- [ ] **AC-015**: Given 缓存命中，When 系统获取指标数据，Then 响应时间 < 10ms
- [ ] **AC-016**: Given 用户缩放操作，When 系统加载新数据，Then 加载过程中不阻塞UI（异步加载）

### 8.6 边界情况

- [ ] **AC-017**: Given 股票历史数据不足283天，When 系统加载数据，Then 尽可能加载所有可用数据，并在可用数据范围内计算指标
- [ ] **AC-018**: Given 股票历史数据完全不存在，When 用户选择该股票，Then 显示错误提示"该股票数据不足"
- [ ] **AC-019**: Given 用户左滑到数据库最早日期，When 查看指标，Then 显示"数据最早日期"的提示，前置指标显示为0
