# 指标预加载与缓存优化需求

## 1. 背景与价值

**为什么做**：当前实战页面中，MACD、KDJ等技术指标以及5日线、10日线、30日线在图表最左侧显示为横线或空白，原因是：
- 指标计算时缺少前置预热数据
- 均线的计算也是动态的，没有提前预计算
- 缩小或左滑K线时，指标无法从最早数据点正常显示

**做成什么样**：实现三层数据加载策略，确保用户缩放或左滑到最左侧时，所有技术指标都能无断档地正常显示。

**关联**：
- 现有指标计算逻辑（`indicator_calculator.dart`）
- 现有预计算逻辑（`battle_provider.dart`）
- K线图表交互优化需求文档

## 3. 三层数据加载策略

### 3.1 数据分层结构

```
┌────────────────────────────────────────────────────────────────────────────┐
│ 第一层：训练日期范围（150天）                                                │
│ 2020-01-01 ~ 2020-05-30                                                    │
│                                                                            │
│ ┌────────────────────────────────────────────────────────────────────┐    │
│ │ 第二层：预加载数据（100天，支持左滑和缩放展示）                          │    │
│ │ 展示范围：2020-01-01 - 100天 ~ 2020-01-01                            │    │
│ │          = 2019-10-01 ~ 2020-01-01                                  │    │
│ │                                                                            │    │
│ │ ┌──────────────────────────────────────────────────────────────┐    │    │
│ │ │ 第三层：指标计算前置数据（33天）                                    │    │    │
│ │ │ 指标前置：2019-10-01 - 33天 = 2019-09-01                        │    │    │
│ │ │ 用于计算预加载数据前33天的MACD、均线等指标                         │    │    │
│ │ └──────────────────────────────────────────────────────────────┘    │    │
│ └────────────────────────────────────────────────────────────────────┘    │
│                                                                            │
│ 最终加载范围：2019-09-01 ~ 2020-05-30                                       │
└────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 各层数据职责

| 层级 | 数据范围 | 天数 | 职责 | 指标依赖 |
|------|---------|------|------|---------|
| **第一层** | 训练日期范围 | 150天 | 训练周期，用户主要操作区域 | - |
| **第二层** | 训练起始 - 100天 | 100天 | 支持用户左滑和缩放查看更多历史数据 | 需要第三层数据计算指标 |
| **第三层** | 第二层起始 - 33天 | 33天 | 为第二层数据提供指标计算前置数据 | 无（纯前置数据） |

### 3.3 指标前置数据需求

| 指标 | 公式 | 最小前置天数 | 说明 |
|------|------|-------------|------|
| **MA5** | 简单移动平均 | 5天 | 最近5天收盘价均值 |
| **MA10** | 简单移动平均 | 10天 | 最近10天收盘价均值 |
| **MA30** | 简单移动平均 | 30天 | 最近30天收盘价均值 |
| **MACD** | EMA(12,26,9) | 33天 | EMA(12)需要12天，EMA(26)需要26天，DEA需要9天，取最大值 |
| **KDJ** | RSV(9,3,3) | 9天 | RSV需要9天，K/D/J各需3天平滑 |
| **RSI** | RSI(14) | 14天 | 14天周期 |
| **BOLL** | MA(20)+2*SD(20) | 20天 | 布林带中轨是20日均线 |
| **DMI** | ADX(14) | 14天 | 需要14天计算基础DM值 |
| **CCI** | CCI(14) | 14天 | 典型周期14天 |
| **WR** | Williams %R | 14天 | 回溯周期14天 |
| **OBV** | 累计成交量 | 1天 | 只需当日数据 |
| **DMA** | DIF(10)+AMA(10) | 10天 | 10日 DIF 和 AMA |
| **BBI** | (MA3+MA6+MA12+MA24)/4 | 24天 | 最长周期是24日 |

**最大前置需求**：**33天**（MACD指标）

### 3.4 数据加载策略

**核心原则**：前置的K线数据必须从数据库真实获取，指标在完整数据上真实计算，不需要填充0

**数据加载策略**：
```
训练起始日期：2020-01-01
需要加载的数据范围：2019-12-01 至 训练结束日期
实际加载天数：训练周期（150天）+ 预加载（100天）+ 指标前置（33天）

最终加载范围：2019-09-01 ~ 2020-05-30（总共283天）

关键点：
1. 所有283天数据都是从数据库真实获取的，不是填充的0
2. 指标在这283天完整数据上预计算
3. 因此用户缩放或左滑到K线图最左侧时，指标曲线从最早数据点开始就是真实计算值
4. 唯一可能出现0值的情况：用户左滑到数据库最早日期（数据不足33天时）
```

**前置填充策略**（仅在数据不足时使用）：
- 如果加载的K线数据不足指标计算所需的前置天数（如用户左滑到数据库最早日期）
- 系统才会使用0填充不足的部分
- 这是边界情况，正常情况下所有指标都是真实计算的

## 4. 按需加载策略

### 4.1 动态加载逻辑

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 用户进入实战页面                                          │
│    ↓                                                        │
│ 2. 加载初始数据（第一层 + 第二层 + 第三层）                   │
│    训练周期（150天）+ 预加载（100天）+ 指标前置（33天）        │
│    总计：283天                                               │
│    ↓                                                        │
│ 3. 用户缩放或左滑                                           │
│    ↓                                                        │
│ 4. 计算新数据范围                                           │
│    newStart = min(当前可见起始日期 - 33天指标前置, 数据库最早)  │
│    newEnd = 训练结束日期                                      │
│    ↓                                                        │
│ 5. 检查缓存                                                 │
│    缓存命中？→ 使用缓存                                       │
│    缓存未命中？→ 查询数据库 → 预计算指标 → 更新缓存           │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 加载示例

**示例 1：用户缩放到 700 根K线**

```
初始状态：
- 用户缩放到 700 根K线
- 可见范围：2019-01-01 ~ 2020-05-30
- 所需数据：2019-01-01 - 33天 ~ 2020-05-30
- 总计：783天（700天可见 + 33天指标前置 + 50天训练起始前）

加载范围：2018-12-01 ~ 2020-05-30（783天）
```

**示例 2：用户缩放到 100 根K线**

```
初始状态：
- 用户缩放到 100 根K线
- 可见范围：2020-01-01 ~ 2020-05-30
- 所需数据：2020-01-01 - 33天 ~ 2020-05-30
- 总计：183天（100天可见 + 33天指标前置 + 50天训练起始前）

加载范围：2019-10-01 ~ 2020-05-30（283天）
```

### 4.3 前端缓存机制

```
缓存键策略：
- 缓存key = "{symbol}_{dataLength}"
- 例如：600519.XSHG_783

缓存结构：
{
  "600519.XSHG_783": {
    "dataLength": 783,
    "startDate": "2018-12-01",
    "endDate": "2020-05-30",
    "klineData": [...],
    "indicators": {...},
    "createdAt": "2020-05-20T10:30:00"
  }
}

缓存淘汰策略：
- LRU缓存，容量50个
- 相同数据长度使用相同缓存
- 缓存有效期：当前会话内有效

缓存更新策略：
- 用户缩放导致数据长度变化时，加载新数据并更新缓存
- 不持久化到本地存储
```

## 5. 数据流设计

### 5.1 初始化流程

```dart
Future<void> _initializeWithData() async {
  // 1. 获取训练日期范围
  final trainingStart = state.trainingStartDate;
  final trainingEnd = state.trainingEndDate;

  // 2. 计算初始加载范围
  final preloadDays = 100; // 预加载数据
  final indicatorPreloadDays = 33; // 指标前置数据

  final dataLoadStart = trainingStart.subtract(
    Duration(days: preloadDays + indicatorPreloadDays),
  );

  // 3. 计算初始可见范围（100根K线）
  final defaultVisibleCount = 100;
  final visibleStart = trainingStart;
  final visibleEnd = trainingEnd;

  // 4. 检查缓存
  final cacheKey = _generateCacheKey(symbol, defaultVisibleCount);
  if (_indicatorCache.containsKey(cacheKey)) {
    _applyCachedData(cacheKey);
    return;
  }

  // 5. 从数据库加载数据
  final extendedData = await _loadExtendedKlineData(
    symbol: symbol,
    startDate: dataLoadStart,
    endDate: trainingEnd,
  );

  // 6. 在扩展数据上预计算所有指标
  final indicators = _precomputeAllIndicators(extendedData);

  // 7. 存储到缓存
  _indicatorCache.put(cacheKey, IndicatorCache(
    dataLength: extendedData.length,
    klineData: extendedData,
    indicators: indicators,
  ));

  // 8. 更新状态
  state = state.copyWith(
    allKlineData: extendedData,
    precomputedIndicators: indicators,
  );
}
```

### 5.2 缩放/滑动时加载流程

```dart
Future<void> _loadDataForRange({
  required int visibleKlineCount,
  required DateTime visibleStart,
  required DateTime visibleEnd,
}) async {
  // 1. 计算需要的数据范围
  final indicatorPreloadDays = 33;
  final dataStart = visibleStart.subtract(
    Duration(days: indicatorPreloadDays),
  );

  // 2. 检查缓存
  final cacheKey = _generateCacheKey(symbol, visibleKlineCount);
  final cachedData = _indicatorCache.get(cacheKey);

  if (cachedData != null && _coversRange(cachedData, dataStart, visibleEnd)) {
    // 缓存命中且覆盖范围，直接使用
    _applyCachedData(cacheKey);
    return;
  }

  // 3. 缓存未命中，加载新数据
  final extendedData = await _loadExtendedKlineData(
    symbol: symbol,
    startDate: dataStart,
    endDate: visibleEnd,
  );

  // 4. 预计算指标
  final indicators = _precomputeAllIndicators(extendedData);

  // 5. 更新缓存
  _indicatorCache.put(cacheKey, IndicatorCache(
    dataLength: extendedData.length,
    klineData: extendedData,
    indicators: indicators,
  ));

  // 6. 更新状态
  state = state.copyWith(
    allKlineData: extendedData,
    precomputedIndicators: indicators,
  );
}
```

### 5.3 缓存数据结构

```dart
class IndicatorCache {
  final String cacheKey;
  final int dataLength;
  final DateTime startDate;
  final DateTime endDate;
  final List<KlineModel> klineData;
  final PrecomputedIndicators indicators;
  final DateTime createdAt;
}

class PrecomputedIndicators {
  final List<MacdData> macd;
  final List<KdjData> kdj;
  final List<double> rsi;
  final List<BollData> boll;
  final List<double> ma5;
  final List<double> ma10;
  final List<double> ma30;
  // ... 其他指标
}

class IndicatorCacheManager {
  static const int maxCacheSize = 50;
  final LinkedHashMap<String, IndicatorCache> _cache = LinkedHashMap();

  void put(String key, IndicatorCache cache);
  IndicatorCache? get(String key);
  bool _coversRange(IndicatorCache cache, DateTime start, DateTime end);
  void remove(String key);
  void clear();
  void clearBySymbol(String symbol);
}
```

## 6. 验收标准

### 6.1 三层数据加载

- [ ] **AC-001**: Given 用户进入实战页面，When 系统加载K线数据，Then 自动加载训练周期（150天）+ 预加载数据（100天）+ 指标前置数据（33天）= 总计283天
- [ ] **AC-002**: Given 用户缩放到700根K线，When 系统加载数据，Then 自动加载700根K线 + 33天指标前置数据
- [ ] **AC-003**: Given 用户缩放到100根K线，When 系统加载数据，Then 自动加载100根K线 + 33天指标前置数据

### 6.2 指标显示

- [ ] **AC-004**: Given K线图缩小或左滑到最左侧，When 用户查看指标，Then MA5、MA10、MA30从最早可见K线开始正常显示曲线（前33天可能显示为0或折线）
- [ ] **AC-005**: Given K线图缩小或左滑到最左侧，When 用户查看指标，Then MACD指标从最早可见K线开始正常显示（不是横线或空白）
- [ ] **AC-006**: Given K线图缩小或左滑到最左侧，When 用户查看指标，Then KDJ、RSI、布林带等其他指标也从最早可见K线开始正常显示
- [ ] **AC-007**: Given 用户在任意缩放级别左滑到最左侧，When 查看指标，Then 所有指标都无断档地正常显示

### 6.3 缓存机制

- [ ] **AC-008**: Given 相同股票第二次缩放到相同级别，When 系统加载数据，Then 使用缓存数据，不重复查询数据库
- [ ] **AC-009**: Given 不同股票进入训练，When 系统加载数据，Then 清除旧缓存，加载新数据
- [ ] **AC-010**: Given 缓存数据达到50个，When 新增缓存，Then 自动清除最旧的缓存项

### 6.4 性能要求

- [ ] **AC-011**: Given 首次加载数据，When 系统查询数据库，Then 响应时间 < 300ms（283天数据）
- [ ] **AC-012**: Given 缓存命中，When 系统获取指标数据，Then 响应时间 < 10ms
- [ ] **AC-013**: Given 用户缩放操作，When 系统加载新数据，Then 加载过程中不阻塞UI（异步加载）

### 6.5 边界情况

- [ ] **AC-014**: Given 股票历史数据不足283天，When 系统加载数据，Then 尽可能加载所有可用数据，并在可用数据范围内计算指标
- [ ] **AC-015**: Given 股票历史数据完全不存在，When 用户选择该股票，Then 显示错误提示"该股票数据不足"
- [ ] **AC-016**: Given 用户左滑到数据库最早日期，When 查看指标，Then 显示"数据最早日期"的提示，前置指标显示为0

## 7. 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-001 | 三层数据加载策略 | 训练周期（150天）+ 预加载（100天）+ 指标前置（33天） |
| BR-002 | 指标前置数据固定为33天 | MACD指标的最大前置需求 |
| BR-003 | 按需动态加载 | 根据用户缩放级别动态决定加载数据量 |
| BR-004 | 缓存以数据长度为键 | 相同缩放级别使用相同缓存 |
| BR-005 | 缓存容量限制为50个 | 防止内存溢出 |
| BR-006 | 前置指标填充0 | 指标前置数据不足33天时，指标值填充0 |
