## 1. 设计理念总览

### 1.1 核心理念

**三大原则**：
1. **分层解耦** — 清晰的分层架构，降低耦合度
2. **职责单一** — 每个模块只做一件事，做好一件事
3. **数据驱动** — 以数据为中心，格式统一可追溯

### 1.2 理念来源

本设计理念基于以下实践经验的总结：

| 经验来源 | 核心教训 | 落地措施 |
|---------|---------|---------|
| 实战页面重构（v7.0） | 大文件(4500+行)难以维护 | 文件行数限制、职责拆分 |
| 状态管理混乱 | 状态分散导致bug难以定位 | Riverpod统一状态管理 |
| 数据格式不统一 | 股票代码格式混用导致错误 | 统一格式转换工具 |
| 缺乏错误处理 | 异常未捕获导致崩溃 | 分层异常处理机制 |

## 2. 核心理念详解

### 2.1 分层解耦

**问题**：
- 早期实战页面将状态管理、业务逻辑、UI渲染全部放在一个4500+行的文件中
- 修改任何功能都需要在大文件中搜索定位
- 难以进行单元测试和并行开发

**解决方案**：
```
┌──────────────────────────────────────────────────────────┐
│ UI Layer (视图层)                                        │
│ 职责：渲染界面、捕获用户交互                             │
│ 特点：最上层，不依赖其他层                               │
├──────────────────────────────────────────────────────────┤
│ Provider Layer (状态层)                                  │
│ 职责：状态管理、事件分发                                  │
│ 特点：管理应用状态，触发业务逻辑                         │
├──────────────────────────────────────────────────────────┤
│ Service Layer (服务层)                                   │
│ 职责：业务逻辑、数据组装                                  │
│ 特点：纯业务逻辑，不涉及UI                               │
├──────────────────────────────────────────────────────────┤
│ Data Layer (数据层)                                      │
│ 职责：数据访问、持久化                                   │
│ 特点：最底层，被其他层依赖                               │
└──────────────────────────────────────────────────────────┘
```

**实施要点**：
- 上层可以依赖下层，下层不能依赖上层
- 跨层依赖通过依赖注入（Provider）实现
- 每层有明确的职责边界

**验证方法**：
```bash
# 检查是否存在跨层依赖
grep -r "import.*service" lib/features/*/providers/
grep -r "import.*provider" lib/features/*/services/
```

### 2.2 职责单一

**问题**：
- 一个方法做太多事情，难以理解和测试
- 一个文件包含太多职责，难以定位问题
- 一个类承担多个角色，违反SRP原则

**解决方案**：
- **方法级**：每个方法不超过50行
- **文件级**：每个文件不超过400行
- **类级**：每个类只有一个主要职责

**具体限制**：

| 类型 | 最大限制 | 拆分信号 |
|------|---------|---------|
| Screen文件 | 400行 | build方法过长 |
| Provider文件 | 300行 | 状态管理逻辑过多 |
| Service文件 | 300行 | 业务逻辑过多 |
| State文件 | 200行 | 状态字段过多 |
| Widget文件 | 150行 | 组件过于复杂 |
| Config文件 | 100行 | 包含非配置代码 |
| 单个方法 | 50行 | 包含多种操作 |

**拆分策略**：

```
┌─────────────────────────────────────────────────────┐
│ 原始文件（违反SRP）                                 │
│ ┌─────────────────────────────────────────────────┐│
│ │ class BigScreen {                               ││
│ │   // 5000行，职责混杂                           ││
│ │ }                                               ││
│ └─────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────┘
                      ↓ 拆分
┌─────────────────────────────────────────────────────┐
│ 拆分后（职责单一）                                   │
│ ┌──────────────┐  ┌──────────────┐                 │
│ │Screen(400行) │  │ State(200行) │                 │
│ │ - build()   │  │ - 状态定义   │                 │
│ │ - 简单回调  │  │ - copyWith() │                 │
│ └──────────────┘  └──────────────┘                 │
│ ┌──────────────┐  ┌──────────────┐                 │
│ │Provider     │  │ Service      │                 │
│ │ (300行)     │  │ (300行)     │                 │
│ │ - 状态管理  │  │ - 业务逻辑  │                 │
│ │ - 事件分发  │  │ - 数据转换  │                 │
│ └──────────────┘  └──────────────┘                 │
│ ┌──────────────┐  ┌──────────────┐                 │
│ │ Widget A    │  │ Widget B    │                 │
│ │ (100行)     │  │ (100行)     │                 │
│ └──────────────┘  └──────────────┘                 │
└─────────────────────────────────────────────────────┘
```

### 2.3 数据驱动

**问题**：
- 股票代码格式不统一（`600519` vs `SH600519` vs `XSHE`）
- 日期时间格式混用（`2023-03-31` vs `2023-03-31T00:00:00`）
- 前后端数据格式不一致导致转换错误

**解决方案**：

#### 2.3.1 数据格式统一原则

**核心原则**：**以数据库存储格式为准**，前端和后端需要做好格式转换

```
数据库（数据源）
    ↓
Repository/DAO（格式转换层）
    ↓
Provider/Service（使用数据库格式）
    ↓
UI（显示前转换）
```

**各层级职责**：

| 层级 | 职责 | 数据格式 |
|------|------|---------|
| **数据库** | 存储数据 | 原始格式 |
| **DAO/Repository** | 格式转换 | 数据库格式 ↔ 领域格式 |
| **Service** | 业务逻辑 | 领域格式 |
| **Provider** | 状态管理 | 领域格式 |
| **UI** | 展示数据 | 显示格式 |

#### 2.3.2 格式转换工具

```dart
// 必须提供双向转换方法
class DataConverter {
  // 领域格式 → 存储格式
  static String toStorage(DomainModel model) {
    return model.toDbFormat();
  }

  // 存储格式 → 领域格式
  static DomainModel fromStorage(String storage) {
    return DomainModel.fromDbFormat(storage);
  }

  // 领域格式 → 显示格式
  static String toDisplay(DomainModel model) {
    return model.toDisplayFormat();
  }

  // 显示格式 → 领域格式
  static DomainModel fromDisplay(String display) {
    return DomainModel.fromDisplayFormat(display);
  }
}
```

#### 2.3.3 禁止的实践

**❌ 直接在UI层转换格式**：
```dart
// 错误：UI层不应该知道数据格式细节
Text(stock.symbol.replaceFirst('SH', ''))  // ❌
```

**✅ 在转换层完成格式转换**：
```dart
// 正确：UI层只关心显示
Text(displaySymbol)  // ✅
```

**❌ 在多个地方重复转换逻辑**：
```dart
// 错误：重复代码
if (symbol.startsWith('SH')) symbol = symbol.substring(2);
```

**✅ 在工具类中统一转换**：
```dart
// 正确：统一的转换逻辑
String displaySymbol = SymbolConverter.toDisplay(symbol, marketCode);
```

## 3. 设计原则

### 3.1 向上抽象

**含义**：向调用方暴露简洁的接口，隐藏内部复杂度

**示例**：

```dart
// ❌ 暴露内部细节
class BattleProvider {
  Future<void> loadStock(String symbol, DateTime? date) async {
    // 内部调用多个Service
    final stockInfo = await stockService.loadStock(symbol);
    final klineData = await klineService.loadKline(symbol, date);
    // 内部处理各种边界情况
    ...
  }
}

// ✅ 暴露简洁接口
class BattleProvider {
  Future<void> initializeWithSymbol({
    required String symbol,
    String? name,
    String? marketCode,
    DateTime? startDate,
  }) async {
    // 内部逻辑对调用方透明
    await _loadKlineData(symbol, startDate);
  }
}
```

**好处**：
- 调用方无需了解内部实现
- 内部重构不影响外部接口
- 易于理解和测试

### 3.2 向下依赖

**含义**：每层只依赖其下层的抽象，不依赖具体实现

**示例**：

```dart
// ✅ 依赖抽象
class StockService {
  final KlineRepository _repository;  // 接口/抽象类

  StockService({required KlineRepository repository})
      : _repository = repository;
}

// ❌ 依赖具体实现
class StockService {
  final KlineDao _dao;  // 具体类，不灵活

  StockService() : _dao = KlineDao();  // 硬编码，无法替换
}
```

**好处**：
- 便于单元测试（可以注入Mock对象）
- 提高代码灵活性
- 降低模块间耦合

### 3.3 向外扩展

**含义**：通过扩展而非修改来增加功能

**示例**：

```dart
// ❌ 通过修改原类增加功能
class BattleState {
  final String newField;  // 修改原有类
}

// ✅ 通过扩展增加功能
class BattleState {
  // 原有字段不变
}

// 新增扩展文件
extension BattleStateExtension on BattleState {
  String get newComputedProperty => doSomething();
}
```

**好处**：
- 不破坏现有代码
- 降低引入bug的风险
- 便于功能演进

## 5. 常见问题与解决方案

### 5.1 如何决定是否需要拆分文件？

**判断标准**：
1. 文件是否超过400行？
2. 文件是否承担多个职责？
3. 文件是否难以理解或测试？
4. 是否有重复代码？

**拆分步骤**：
1. 识别文件中的不同职责
2. 将每种职责抽取为独立模块
3. 通过依赖注入建立联系
4. 确保拆分后功能不变

### 5.2 如何处理状态持久化？

**策略选择**：

| 数据类型 | 持久化策略 | 示例 |
|---------|-----------|------|
| 临时状态 | 仅内存 | 列表筛选条件 |
| 用户偏好 | SharedPreferences | 主题选择、语言 |
| 重要数据 | 数据库 | 训练记录、账户信息 |
| 缓存数据 | 内存+磁盘 | K线历史数据 |

**实现模式**：
```dart
class BattleProvider extends _$Battle {
  @override
  BattleState build() {
    // 从数据库恢复状态
    _restoreState();
    return _initialState;
  }

  Future<void> _restoreState() async {
    final savedState = await _db.loadState();
    if (savedState != null) {
      _initialState = savedState;
    }
  }

  Future<void> saveState() async {
    // 保存到数据库
    await _db.saveState(state);
  }
}
```

### 5.3 如何处理跨模块依赖？

**原则**：模块间应该低耦合，必要时通过接口解耦

**解决方案**：

```dart
// 方案1：通过Provider注入接口
final stockServiceProvider = Provider<StockService>((ref) {
  return StockService();
});

// 方案2：通过参数传递
class TrainingService {
  final StockService stockService;
  final DatabaseService dbService;

  TrainingService({
    required this.stockService,
    required this.dbService,
  });
}

// 方案3：通过组合模式
class BattleProvider {
  late final StockService _stockService;
  late final TrainingService _trainingService;

  BattleProvider(Ref ref) {
    _stockService = ref.read(stockServiceProvider);
    _trainingService = TrainingService(stockService: _stockService);
  }
}
```

### 5.4 如何处理错误和异常？

**分层处理策略**：

```
UI层：展示错误
  ↓
Provider层：转换错误为用户消息
  ↓
Service层：捕获并记录错误
  ↓
DAO层：捕获基础设施错误
```

**实现示例**：

```dart
// Service层
Future<void> doSomething() async {
  try {
    await _repository.doSomething();
  } catch (e, stackTrace) {
    _logger.e('Operation failed', e, stackTrace);
    rethrow ServiceException('操作失败，请重试');
  }
}

// Provider层
Future<void> handleOperation() async {
  try {
    state = state.copyWith(isLoading: true);
    await _service.doSomething();
    state = state.copyWith(isLoading: false, isSuccess: true);
  } on ServiceException catch (e) {
    state = state.copyWith(isLoading: false, errorMessage: e.userMessage);
  }
}
```

### 5.5 如何保证数据一致性？

**核心原则**：
1. 数据只在一处定义
2. 格式转换只在一处进行
3. 状态更新使用不可变模式

**实践要点**：
```dart
// ✅ 正确：使用copyWith进行不可变更新
state = state.copyWith(
  items: newItems,
  currentIndex: newIndex,
);

// ❌ 错误：直接修改状态
state.items.add(newItem);
state.currentIndex = newIndex;

// ✅ 正确：在转换层统一格式
String displaySymbol = SymbolConverter.toDisplay(dbSymbol, marketCode);

// ❌ 错误：在多个地方重复格式转换
String symbol = symbol.startsWith('SH') ? symbol.substring(2) : symbol;
```

## 7. 术语表

| 术语 | 定义 | 示例 |
|------|------|------|
| **State** | 状态模型，表示应用在某一时刻的数据 | `BattleState` |
| **Provider** | Riverpod状态管理器 | `BattleProvider` |
| **Notifier** | 状态变更通知器 | `BattleNotifier` |
| **Service** | 业务逻辑服务 | `StockService` |
| **Repository** | 数据仓库，封装数据访问 | `KlineRepository` |
| **DAO** | 数据库访问对象 | `KlineDao` |
| **Widget** | Flutter UI组件 | `StockInfoBar` |
| **Screen** | 页面级Widget | `BattleScreen` |
| **copyWith** | 不可变对象更新方法 | `state.copyWith(x: 1)` |
| **Consumer** | Riverpod状态消费者 | `Consumer<T>` |
