## 1. 数据库数据同步设计

### 1.1 设计目标

- 从外部数据库同步数据到应用数据库
- 自动处理数据冲突和重复
- 提供示例数据用于开发测试
- 确保数据完整性和一致性

### 1.2 技术架构

```
┌─────────────────────────────────────────────────────────┐
│                    应用启动流程                          │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│             检查外部数据库是否存在                       │
│         assets/data/stock_data/stock_data.db            │
└─────────────────────────────────────────────────────────┘
                │                           │
              存在                          不存在
                │                           │
                ▼                           ▼
┌───────────────────────────┐    ┌───────────────────────────┐
│  从外部数据库读取数据     │    │  初始化示例数据           │
│  StockDataSyncService     │    │  ConfigDao.init*Data     │
└───────────────────────────┘    └───────────────────────────┘
                │                           │
                └──────────┬────────────────┘
                           │
                           ▼
         ┌─────────────────────────────────┐
         │   写入应用数据库                │
         │   ~/Library/.../kline_trainer.db│
         └─────────────────────────────────┘
```

### 1.3 数据同步服务

**文件位置**：`lib/data/services/stock_data_sync_service.dart`

**服务职责**：
- 从外部数据库读取股票数据
- 从外部数据库读取K线数据
- 将数据写入应用数据库
- 处理数据冲突
- 提供数据完整性检查

**核心方法**：

```dart
class StockDataSyncService {
  /// 从外部数据库同步股票数据

  /// 从外部数据库同步K线数据
```

## 3. 数据库架构设计

### 3.1 项目路径 vs 沙箱路径

| 类型 | 路径 | 内容 | 用途 |
|------|------|------|------|
| **项目路径** | `lib/data/database/` | 表定义代码 | 定义数据结构 |
| **运行时路径** | `~/Library/.../Documents/` | 实际数据库文件 | 存储运行时数据 |

### 3.2 iOS模拟器数据库路径

完整路径示例：
```
/Users/[用户名]/Library/Developer/CoreSimulator/Devices/[设备ID]/data/Containers/Data/Application/[应用ID]/Documents/kline_trainer.db
```

**为什么不在项目路径**：
1. **沙箱安全机制**：iOS/macOS应用使用沙箱机制，数据必须存储在特定目录
2. **数据隔离**：每个应用独立存储，互不干扰
3. **数据持久化**：应用关闭后数据仍然保留
4. **数据清理**：删除应用时自动清理数据，释放空间

### 3.3 数据库表结构

#### 3.3.1 symbols表（股票列表）

```sql
CREATE TABLE symbols (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    symbol TEXT NOT NULL UNIQUE,
    name TEXT,
    market_code TEXT,
    sector TEXT,
    enabled INTEGER DEFAULT 1,
    last_price REAL,
    change REAL,
    created_at TEXT,
    updated_at TEXT
);
```

#### 3.3.2 kline_data表（K线数据）

```sql
CREATE TABLE kline_data (
    symbol TEXT NOT NULL,
    market_code TEXT NOT NULL,
    period TEXT NOT NULL,
    trade_date INTEGER NOT NULL,
    open REAL NOT NULL,
    close REAL NOT NULL,
    high REAL NOT NULL,
    low REAL NOT NULL,
    volume REAL NOT NULL,
    amount REAL NOT NULL,
    turnover_rate REAL,
    pe REAL,
    pb REAL,
    created_at INTEGER NOT NULL,
    PRIMARY KEY (symbol, period, trade_date)
);
```

## 5. 选股日期优化

### 5.1 问题分析

之前选股时使用当前日期，但可能没有对应日期的K线数据，导致选股结果为0。

### 5.2 解决方案

获取有K线数据的最近日期作为选股日期。

**实现位置**：`lib/data/database/daos/stock_filter_dao.dart`

**方法实现**：
```dart
Future<DateTime?> getLatestKlineDate() async {
  final result = await (selectOnly(klineData)
        ..addColumns([klineData.tradeDate.max()])
        ..where(klineData.period.equals('day')))
      .getSingle();
  
  final timestamp = result.read(klineData.tradeDate.max());
  if (timestamp == null) return null;
  
  return DateTime.fromMillisecondsSinceEpoch(timestamp * 1000);
}
```

**使用位置**：`lib/data/repositories/stock_filter_repository.dart`

```sql
SELECT 
    symbol, 
    market_code, 
    datetime(trade_date, 'unixepoch') as trade_date,
    printf('%.2f', open) as open,
    printf('%.2f', close) as close,
    high,
    low,
    volume
```

## 8. 性能优化建议

1. **批量插入**：使用batch操作减少数据库事务次数
2. **索引优化**：在频繁查询的字段上创建索引
3. **缓存机制**：选股结果使用缓存避免重复计算
4. **异步加载**：使用Future和async/await处理耗时操作
5. **分页查询**：大数据量时使用分页避免内存问题
