## 4. 模块划分原则

1. **按业务功能划分**：将相关功能聚合成独立模块
2. **单一职责**：每个文件/类只负责一个功能
3. **高内聚低耦合**：模块内部紧密相关，模块间依赖最小化
4. **可测试性**：便于单元测试和集成测试
5. **可扩展性**：便于后续功能扩展和维护

## 5. 命名规范

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 文件 | 小写+下划线 | `kline_chart_screen.dart` |
| 类 | 大驼峰 | `KlineChartScreen` |
| 函数/方法 | 小驼峰 | `fetchKlineData()` |
| 变量 | 小驼峰 | `klineData` |
| 常量 | 全大写+下划线 | `BASE_URL` |
| Provider | 小驼峰 + Provider后缀 | `klineDataProvider` |

## 6. 架构设计

### 6.1 分层架构

```
┌─────────────────────────────────────────────┐
│              UI Layer (Features)           │
│  - KlineChartScreen, CourseListScreen...   │
├─────────────────────────────────────────────┤
│           Provider Layer (Riverpod)        │
│  - AuthProvider, KlineProvider...          │
├─────────────────────────────────────────────┤
│         Repository Layer                   │
│  - KlineRepository, CourseRepository...    │
├─────────────────────────────────────────────┤
│             API Layer                      │
│  - KlineApi, CourseApi, AuthApi...         │
├─────────────────────────────────────────────┤
│          Data Source Layer                 │
│  - SQLite (Drift), REST API, WebSocket    │
└─────────────────────────────────────────────┘
```

### 6.2 数据流

```
UI → Provider → Repository → API → Data Source
     ↓           ↓            ↓        ↓
    State     Business     Network   Database
              Logic        Request   Operations
```

### 6.3 核心组件关系

| 组件 | 职责 | 依赖 |
|------|------|------|
| `main.dart` | 应用入口 | `ProviderScope`, `dotenv` |
| `app.dart` | 根组件 | `MaterialApp`, `GoRouter` |
| `AppRoutes` | 路由管理 | `GoRouter`, 各Screen |
| `Providers` | 状态管理 | `Riverpod`, Repositories |
| `Repositories` | 数据抽象 | APIs, Database |
| `APIs` | 网络请求 | `Dio` |
