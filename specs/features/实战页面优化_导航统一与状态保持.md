# 实战页面优化 — 功能需求说明书

## 1. 背景与价值

**为什么做**：当前实战页面存在三个核心问题：
1. 每个页面都有独立的底部导航栏，状态无法保持，用户切换导航后返回实战页面会丢失训练进度
2. 直接进入实战页面时使用硬编码的默认股票，无法支持完全随机的股票训练
3. 没有从首页选股跳转时，实战页面无法独立初始化并支持训练

**做成什么样**：
- 统一底部导航栏架构，使用IndexedStack保持所有Tab页状态
- 实战页面独立初始化时，从数据库随机选择有完整K线数据的股票
- 用户在任意Tab切换后返回，都能无缝继续训练，所有状态完整保留
- 训练周期固定150天，展示训练前60天K线用于指标预热

**关联**：项目核心功能优化，涉及首页、实战、记录、我的四个Tab页的导航架构重构

---

## 2. 范围

### 本次做

- **导航架构重构**：统一底部导航栏，将首页、实战、记录、我的四个页面整合为统一的Tab结构
- **状态保持机制**：使用IndexedStack + AutomaticKeepAliveClientMixin确保所有Tab页状态保持
- **随机选股初始化**：实战页面独立初始化时，从数据库随机选择有完整K线数据的股票
- **全量状态保持**：返回实战页面时保持K线位置、账户信息、交易记录、指标参数等所有状态
- **修复多处导航栏**：移除实战页面内嵌的独立导航栏，确保只有一处底部导航栏

### 本次不做

- 不修改首页选股功能逻辑
- 不修改K线图表渲染逻辑
- 不修改交易功能逻辑
- 不修改数据库结构
- 不添加新的功能模块

## 4. 需求详述

### 4.1 核心流程

#### 流程A：直接进入实战页面（无选股跳转）

```
1. 用户点击底部导航栏"实战"
       ↓
2. 系统检查是否为首次进入实战页面
       ↓
3. 若是首次进入且无路由参数：
   - 查询数据库获取有完整K线数据的股票列表
   - 随机选择一只股票
   - 计算训练周期：训练150天 + 前置60天K线
   - 加载该股票的K线数据
   - 初始化训练状态（资金10万、空仓）
       ↓
4. 进入训练界面，显示K线图表和随机股票信息
```

#### 流程B：从首页选股后跳转实战页面

```
1. 用户在首页选择选股条件和时间范围
       ↓
2. 点击确认按钮，跳转到实战页面
       ↓
3. 实战页面接收路由参数（symbol, name, market, date等）
       ↓
4. 加载指定的股票K线数据
       ↓
5. 进入训练界面，显示指定股票信息
```

#### 流程C：Tab切换后返回实战页面

```
1. 用户在实战页面进行训练
       ↓
2. 用户点击底部导航栏切换到其他Tab（如"记录"或"我的"）
       ↓
3. IndexedStack隐藏实战页面（但保留状态）
       ↓
4. 显示目标Tab页面
       ↓
5. 用户点击"实战"Tab返回
       ↓
6. IndexedStack显示实战页面
       ↓
7. 实战页面从保持的状态恢复：
   - K线位置（当前第几天）
   - 账户信息（资金、持仓、成本）
   - 交易记录（所有买卖点位）
   - 指标参数（当前选中的指标类型）
   - 当前股票信息
```

### 4.2 随机选股逻辑

#### 4.2.1 随机选股条件

股票必须满足以下条件才能被随机选中：
1. **数据库有数据**：该股票在K线数据库中有至少210天的日线数据（150天训练 + 60天预热）
2. **市场限定**：仅限A股市场（SH/SZ开头的股票代码）
3. **启用状态**：股票状态为enabled=true

#### 4.2.2 随机算法

1. 从数据库查询满足条件的股票列表
2. 使用随机数生成器选择一只股票
3. 如果选择的股票数据不足，自动重试（最多3次）
4. 如果所有股票都失败，使用保底方案（深科技SZ000021）

#### 4.2.3 训练周期配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 训练天数 | 150天 | 用户实际进行模拟交易的天数 |
| 前置K线 | 60天 | 用于技术指标计算的数据预热 |
| 总加载天数 | 210天 | 训练天数 + 前置K线 |

### 4.3 状态保持机制

#### 4.3.1 需要保持的状态

| 状态类型 | 具体内容 | 保持方式 |
|---------|---------|---------|
| K线位置 | `_currentDayIndex` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 账户信息 | `_accountBalance`, `_positionQuantity`, `_positionCost`, `_totalProfitLoss` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 交易记录 | `_tradePoints`（买卖点位列表） | StatefulWidget + AutomaticKeepAliveClientMixin |
| 指标参数 | `_selectedPeriod`, `_selectedTopIndicator`, `_selectedBottomIndicator` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 当前股票 | `_currentSymbol`, `_currentSymbolName`, `_currentMarketCode` | StatefulWidget + AutomaticKeepAliveClientMixin |
| K线数据 | `_allKlineData` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 训练日期 | `_trainingStartDate` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 训练周期 | `_trainingDays` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 可见范围 | `_visibleStartIndex`, `_visibleKlineCount`, `_zoomScale` | StatefulWidget + AutomaticKeepAliveClientMixin |
| 训练阶段 | `_trainingPhase`（开盘/收盘） | StatefulWidget + AutomaticKeepAliveClientMixin |
| 重播模式 | `_isReplayMode` | StatefulWidget + AutomaticKeepAliveClientMixin |

#### 4.3.2 状态保持实现方案

**方案：IndexedStack + AutomaticKeepAliveClientMixin**

1. 创建统一的Tab页面容器 `MainTabPage`
2. 使用 `IndexedStack` 管理四个Tab页面（首页、实战、记录、我的）
3. 每个需要保持状态的页面实现 `AutomaticKeepAliveClientMixin`
4. 所有页面共享同一个 `Scaffold` 和 `BottomNavigationBar`

### 4.4 导航架构重构

#### 4.4.1 当前问题

当前有8个地方定义了独立的 `BottomNavigationBar`：
- `home_screen.dart`
- `battle_screen.dart`（实战页面）
- `records_screen.dart`
- `mine_screen.dart`
- `course_list_screen.dart`
- `trading_screen.dart`
- `profile_screen.dart`
- `kline_chart_screen.dart`

#### 4.4.2 重构目标

统一为一个 `BottomNavigationBar`，位于最外层的Tab容器中：
- 实战页面移除内嵌的 `BottomNavigationBar`
- 其他页面如果不在Tab容器中，保留各自的导航栏
- 如果在Tab容器中，则移除各自的导航栏

### 4.5 交互规则

#### 4.5.1 底部导航栏交互

- 点击Tab项立即切换页面
- 当前Tab显示高亮状态（使用 `AppTheme.accent` 颜色）
- 非当前Tab显示普通状态（使用 `AppTheme.muted` 颜色）
- 切换时无动画过渡，立即显示目标页面

#### 4.5.2 实战页面状态恢复

- 返回时立即显示恢复后的状态，无加载过程
- 如果K线数据存在，直接使用缓存数据
- 如果数据为空（如首次进入），执行初始化流程
