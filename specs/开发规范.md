# K线训练营App - 开发规范文档

## 一、代码风格规范

### 1.1 Dart代码规范

#### 1.1.1 格式规则
- 使用 **2** 空格缩进
- 每行代码不超过 **120** 字符
- 使用 **单引号** 包裹字符串，除非需要字符串插值
- 文件末尾保留一个空行

#### 1.1.2 命名规则

| 类型 | 规则 | 示例 |
|------|------|------|
| 类名 | 大驼峰（UpperCamelCase） | `KlineChartScreen` |
| 函数/方法 | 小驼峰（lowerCamelCase） | `fetchKlineData()` |
| 变量/参数 | 小驼峰（lowerCamelCase） | `klineData`, `userId` |
| 常量 | 全大写+下划线 | `BASE_URL`, `MAX_RETRY` |
| 文件 | 小写+下划线 | `kline_chart_screen.dart` |
| 私有成员 | 下划线开头 | `_privateField` |

#### 1.1.3 注释规范
- 公共API必须有文档注释（`///`）
- 复杂逻辑必须有行注释
- 注释语言：中文

### 1.2 Flutter Widget规范

- Widget应尽可能小且职责单一
- 使用 `const` 构造函数（如果可能）
- 避免深层嵌套，使用 `Builder` 或拆分组件
- Widget命名以 `Widget` 结尾（如 `CustomButtonWidget`）

## 二、状态管理规范

### 2.1 Riverpod使用规范

- 使用 **代码生成** 模式（`@riverpod` 注解）
- Provider命名以 `Provider` 结尾（如 `klineProvider`）
- 避免全局状态，使用 `ref.watch` 按需监听
- 复杂状态拆分为多个Provider

### 2.2 状态分层

| 层级 | 职责 |
|------|------|
| UI层 | 展示数据，处理用户交互 |
| Provider层 | 管理业务状态 |
| Repository层 | 封装数据获取逻辑 |
| API层 | 封装HTTP请求 |

## 三、Git工作流规范

### 3.1 分支策略

| 分支 | 用途 |
|------|------|
| `main` | 主分支，生产环境代码 |
| `develop` | 开发分支，整合功能 |
| `feature/*` | 功能分支，开发新功能 |
| `bugfix/*` | Bug修复分支 |
| `hotfix/*` | 紧急修复分支 |

### 3.2 提交规范

使用 **Conventional Commits** 格式：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**type类型**：
- `feat`: 新功能
- `fix`: Bug修复
- `docs`: 文档更新
- `style`: 代码格式（不影响代码逻辑）
- `refactor`: 重构（既不新增功能也不修复Bug）
- `test`: 测试相关
- `chore`: 构建/工具相关

**示例**：
```
feat(kline): 添加MA均线指标
fix(course): 修复课程详情加载失败
docs(readme): 更新项目说明
```

## 四、错误处理规范

### 4.1 异常处理

- 使用 `try-catch` 捕获异常
- 对网络请求添加超时处理
- 使用统一的错误处理类

### 4.2 错误类型

| 类型 | 说明 |
|------|------|
| `NetworkException` | 网络错误 |
| `ServerException` | 服务端错误 |
| `AuthException` | 认证错误 |
| `DataException` | 数据解析错误 |

## 五、测试规范

### 5.1 测试覆盖率

- 单元测试覆盖率 ≥ 70%
- 核心业务逻辑必须有测试
- Widget测试覆盖关键UI流程

### 5.2 测试文件结构

```
test/
├── unit/              # 单元测试
│   ├── providers/     # Provider测试
│   ├── utils/         # 工具函数测试
│   └── repositories/  # 仓库测试
├── widget/            # Widget测试
└── integration/       # 集成测试
```

## 六、资源管理规范

### 6.1 静态资源

- 图片资源放在 `assets/images/`
- 字体资源放在 `assets/fonts/`
- 在 `pubspec.yaml` 中声明资源

### 6.2 国际化

- 使用 `intl` 包进行国际化
- 字符串资源放在 `lib/l10n/`
- 支持中文和英文

## 七、安全规范

### 7.1 敏感信息

- 敏感配置使用环境变量（`.env`）
- 不要将密钥、密码提交到版本控制
- API密钥使用后端代理

### 7.2 数据加密

- 用户密码使用哈希存储
- 敏感数据传输使用HTTPS

## 八、性能优化规范

### 8.1 代码优化

- 使用 `const` 构造函数
- 避免不必要的Widget重建
- 使用 `ListView.builder` 优化列表性能

### 8.2 图片优化

- 使用合适的图片格式（WebP优先）
- 压缩图片大小
- 使用占位图和懒加载

## 十一、技术决策记录

### 11.1 图表库选择

| 选项 | 评估 | 决策 |
|------|------|------|
| Syncfusion Charts | 功能强大但与Flutter 3.41存在API冲突 | ❌ 放弃 |
| fl_chart | 轻量、稳定、兼容性好 | ✅ 采用 |

**理由**: Syncfusion与Flutter 3.41的ThemeData API存在命名冲突（`SelectionDetails`），而fl_chart更轻量且没有兼容性问题。

### 11.2 ThemeData API适配

Flutter 3.17+ 对ThemeData API进行了调整：

| 旧API | 新API |
|-------|-------|
| `accentColor` | `colorScheme.secondary` |
| `backgroundColor` | `colorScheme.background` |
| `FontWeight.semiBold` | `FontWeight.w600` |

### 11.3 macOS构建注意事项

由于GPU着色器编译问题，macOS构建需要特殊环境变量：

```bash
FLUTTER_DISABLE_GPU_SHADERS=true flutter run -d macos --release
```
